## Objective

Restore the frozen Substrate contract after Packet 4 drift:

1. keep continuity-first attach and persisted-truth planning;
2. remove prompt-free fresh attach execution from active runtime paths;
3. fail closed when fresh attach would be required;
4. correct tests and planning docs so the repo no longer overclaims compliance.

## Frozen Inputs From Spec

This plan consumes the following reviewed inputs from [SPEC-31_25.md](./SPEC-31_25.md):

1. 28.5, 29.75, and 30 remain authoritative and disallow blank-prompt control semantics as a forward architecture.
2. Packet 4's persisted host-attach planning is worth keeping where it does not require forbidden runtime behavior.
3. `hidden_owner_helper` may remain as internal launch plumbing if the prompt-free fresh attach regression is removed.
4. No backend currently has a reviewed sanctioned non-prompt fresh attach mechanism for this slice.
5. This is a remediation slice, not a rewrite of slice 31, and it should stand as its own corrective planning set.

## Plan Summary

The repo currently contains two different kinds of Packet 4 work:

1. good work:
   - automatic attach planning now resolves from persisted `HostAttachContract` truth;
   - continuity-versus-fresh need is modeled explicitly;
   - manual and automatic attach share more of the same internal launch plumbing.
2. bad work:
   - control-only attach startup now falls through to prompt-free fresh attach execution when continuity is absent;
   - backend adapters now expose prompt-free fresh attach helpers for Codex and Claude;
   - tests and docs were updated to normalize that behavior.

The remediation should preserve the good work and remove the bad work.

## Exact Gap To Close

1. `async_repl.rs` currently treats "no continuity selector" as permission to execute fresh control-only attach.
2. `prompt_fulfillment.rs` currently contains backend-specific prompt-free fresh attach helpers for both Codex and Claude.
3. Tests currently assert that fresh attach without continuity is valid startup preparation.
4. Historical slice-31 materials currently contain Packet 4 language that conflicts with the frozen floor, but this corrective slice should fix the runtime and carry the corrected truth without depending on rewrites of those artifacts.

## Frozen Execution Contract

If implementation wants to deviate from this contract, revise this plan first.

### Control-only attach contract

1. Control-only attach remains sanctioned only for continuity-backed recovery in this slice.
2. Missing continuity truth does not authorize a fresh control-only launch.
3. When fresh attach would be required, the runtime must fail closed before backend launch with an explicit `unsupported_attach_mode`-style diagnostic.

### Persisted-truth planning contract

1. `HostAttachContract` remains the only durable attach-truth baseline.
2. Automatic attach planning may still determine that continuity is unavailable and fresh would be required.
3. That planning outcome does not itself authorize runtime execution.
4. No path may reconstruct attach truth from stale participant state to escape the fail-closed rule.

### Backend adapter contract

1. Codex and Claude wrappers may expose prompt-bearing launch and continuity-backed attach where already sanctioned.
2. They do not own whether Substrate is allowed to perform prompt-free fresh attach.
3. Any backend helper that would execute prompt-free fresh attach must be removed or made unreachable in this slice.

### Documentation contract

1. This corrective slice owns the corrected truth and stands independently of slice-31 planning artifacts.
2. Future fresh attach work must be explicitly deferred to a new sanctioned slice.
3. The 31.25 corrective closeout is the authoritative shipped-behavior record for this remediation and supersedes Packet 4's prompt-free fresh-attach overclaim.

## Implementation Order

### Packet 1: Remove Fresh Control-Only Runtime Fallback

Goal:

1. stop control-only attach startup from falling through to fresh attach execution;
2. remove backend helpers that implement prompt-free fresh attach;
3. replace fallback with explicit fail-closed diagnostics.

Why first:

1. this is the direct contract violation;
2. leaving it live makes every later doc/test step dishonest;
3. it is the smallest rollback-friendly runtime slice.

Primary touch surface:

1. `crates/shell/src/repl/async_repl.rs`
2. `crates/shell/src/execution/prompt_fulfillment.rs`

Verification checkpoint:

1. no active runtime path can launch prompt-free fresh attach;
2. continuity-backed attach still works;
3. targeted unit coverage reflects the new fail-closed rule.

### Packet 2: Preserve Persisted-Truth Planning While Failing Closed On Fresh-Needed Cases

Goal:

1. keep auto-attach and reattach planning rooted in persisted truth;
2. explicitly classify fresh-needed cases as unsupported in this slice;
3. make manual and automatic attach outcome handling consistent.

Why second:

1. Packet 1 removes the forbidden runtime path, but Packet 2 makes the remaining planner/executor story truthful;
2. it isolates the policy decision from adapter plumbing;
3. it keeps the good Packet 4 work instead of bluntly reverting the whole packet.

Primary touch surface:

1. `crates/shell/src/execution/agent_runtime/auto_attach.rs`
2. `crates/shell/src/execution/agent_runtime/control.rs`
3. `crates/shell/src/execution/agents_cmd.rs`
4. `crates/shell/src/execution/agent_runtime/state_store.rs`

Verification checkpoint:

1. continuity-backed attach still succeeds when allowed;
2. continuity-missing/fresh-needed cases fail closed before backend launch;
3. automatic-attach durable outcomes and manual `reattach` diagnostics stay coherent.

### Packet 3: Correct Tests, Spec Artifacts, And Closeout Truth

Goal:

1. remove tests that encode fresh attach without continuity as valid shipped behavior;
2. write corrected remediation-specific spec/closeout truth without rewriting slice-31 planning artifacts;
3. record the deferred status of any future sanctioned fresh attach work.

Why third:

1. the corrective record should not be written ahead of the runtime change it describes;
2. Packet 1 and 2 determine the exact fail-closed wording and test shapes;
3. this is the integration checkpoint that prevents future confusion.

Primary touch surface:

1. `crates/shell/src/repl/async_repl.rs` tests
2. `llm-last-mile/SPEC-31_25.md`
3. `llm-last-mile/PLAN-31_25.md`
4. `llm-last-mile/TASKS-31_25.md`
5. new remediation closeout artifact under `llm-last-mile/`

Verification checkpoint:

1. remediation docs truthfully describe the restored contract;
2. regression coverage pins the restored contract;
3. validation wall is green.

## Risks And Mitigations

1. Risk: remediation overreacts and discards the persisted-truth planning improvements.
   Mitigation: preserve `auto_attach.rs` planner logic except where it currently authorizes forbidden runtime execution.
2. Risk: runtime behavior becomes fail-closed but operator-facing diagnostics become vague.
   Mitigation: introduce stable explanation-ready errors for "fresh attach required but not sanctioned."
3. Risk: a standalone corrective slice feels disconnected from the original Packet 4 history.
   Mitigation: reference the historical Packet 4 docs as source gap context, but keep the corrected contract in this remediation slice's own artifacts.
4. Risk: backend-specific wrapper affordances creep back in later.
   Mitigation: add regression tests that assert the active control-only path does not use prompt-free fresh attach helpers.
