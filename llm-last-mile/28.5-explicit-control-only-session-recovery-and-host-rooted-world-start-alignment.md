# SOW: Explicit Control-Only Session Recovery And Host-Rooted World-Start Alignment

Status: remaining-work truth-freeze and remediation slice. This SOW follows [28-gateway-mediated-llm-fulfillment-without-lifecycle-regression.md](28-gateway-mediated-llm-fulfillment-without-lifecycle-regression.md) and exists to close the control-plane contract gap exposed by [UAA_PROMPTLESS_RESUME_FORK_SYNTHESIS.md](../UAA_PROMPTLESS_RESUME_FORK_SYNTHESIS.md). It is the architectural bridge between the landed gateway-mediated fulfillment seam and the remaining launch/lifecycle slices in [29-shared-agent-dispatch-envelope-and-capability-override-contract.md](29-shared-agent-dispatch-envelope-and-capability-override-contract.md), [30-public-world-scoped-agent-start-and-capability-flags.md](30-public-world-scoped-agent-start-and-capability-flags.md), and [31-lazy-host-attach-for-host-rooted-world-start.md](31-lazy-host-attach-for-host-rooted-world-start.md).

This slice is not “more gateway work.” The gateway-mediated prompt-fulfillment seam is already the production direction. The missing work is to finalize the cross-repo contract for control-only session recovery, stop treating blank prompt as durable meaning, and freeze the host-rooted world-start direction so the next SOWs are no longer carrying contradictory options.

## Objective

Finalize the Substrate architecture and UAA boundary correction for control-only recovery and successor-session setup so that:

1. Substrate public semantics stay frozen:
   - `start` and `turn` are the only prompt-bearing public verbs,
   - `reattach` is control-only attached-owner recovery,
   - `fork` is control-only successor durable-session allocation,
   - detached-world follow-up remains fail-closed until host ownership is attached through the sanctioned path.
2. Substrate stops depending on blank-prompt `agent_api.session.resume.v1` or blank-prompt `agent_api.session.fork.v1` entirely.
3. Control-only resume/reattach is treated as a Substrate-owned orchestration seam rather than as a UAA v1 semantic.
4. UAA prompt-bearing `resume` and `fork` semantics are restored and kept prompt-bearing by default.
5. The remaining work in `29`, `30`, and `31` can proceed on one closed architectural path:
   - host-rooted public `--scope world`,
   - shared launch plus future-host-attach contract,
   - lazy host attach without synthetic prompts.

## Why This Slice Exists

The current repo state is mismatched in a way that is now materially blocking the next slices:

1. Substrate already treats `reattach` and `fork` as control actions at the public lifecycle layer in [docs/USAGE.md](../docs/USAGE.md) and [HOST_ORCHESTRATOR_INTENDED_BEHAVIOR_TRUTH.md](../HOST_ORCHESTRATOR_INTENDED_BEHAVIOR_TRUTH.md).
2. Live Substrate hidden owner-helper startup still maps `Resume`, `ResumeOneTurn`, and `Fork` through the same `agent_api.session.resume.v1` shaping path and turns `NoPromptRecovery` into `prompt: ""` in [crates/shell/src/repl/async_repl.rs](../crates/shell/src/repl/async_repl.rs).
3. Live UAA code on `feat/promptless-resume` at commit `3b7a4ef` accepts blank prompts for `agent_api.session.resume.v1` and `agent_api.session.fork.v1`, but normative UAA docs still say prompt must be non-empty and still describe resume/fork as prompt-bearing operations.
4. `30` still carries host-rooted versus standalone world-root as an open product decision even though [ADR-0047](../docs/project_management/adrs/draft/ADR-0047-host-orchestrator-durable-session-and-parked-resumable-ownership.md) and the current host durability model already make host-rooted authority the aligned direction.

This SOW exists to close those gaps before more work lands on top of them.

## Front-Running Architectural Shape

The current front-running replacement shape for this seam is to decompose the vague hidden owner-helper model into explicit Substrate-owned responsibilities, then validate that decomposition against the live code and the remaining SOW chain.

The leading candidate shape is:

1. Durable orchestration session
   - Pure Substrate state and authority.
   - Owns durable lifecycle truth, orchestration routing, inbox/task state, lineage, posture, and future attach prerequisites.

2. Attached host execution client
   - The prompt-bearing agent process that can attach, detach, park, and resume.
   - Exists as an execution client of the durable session, not as the durable authority itself.

3. Control-plane attach worker
   - A Substrate-owned recovery/attachment mechanism for `reattach`, with no prompt semantics.
   - Restores or establishes attached host ownership against the durable session without treating the action as a prompt-bearing run.

4. Prompt-turn launcher
   - A distinct path for resumed prompt-bearing work.
   - Owns prompt-taking `turn` behavior and any resumed prompt-bearing host execution startup.

5. Successor allocator
   - A distinct Substrate control path for `fork` lineage/session allocation.
   - Owns successor durable-session creation and lineage truth even when no backend-native fork-without-turn operation exists.

This is not yet declared final architecture truth. It is the current best shape to research and validate or invalidate.

Required stance for this slice:

1. The repo should treat this decomposition as the leading model to test against current code and docs.
2. If research confirms it, later slices should plan and land code against this decomposed model rather than against the current catch-all hidden owner-helper abstraction.
3. SOWs `29`, `30`, and `31` are expected to be updated so they align with this decomposition and the ownership decisions fixed here.
4. Those follow-on SOWs may also be the concrete slices that land parts of the required implementation, so alignment updates are not optional editorial cleanup; they are part of execution planning.
5. If research uncovers a better shape, that better shape must be made explicit and then planned to land in the codebase instead of silently drifting back to the current vague helper seam.
6. No later slice should assume the current hidden owner-helper abstraction is itself the final design just because it exists in live code.

## Frozen Decisions

This slice must make the following decisions explicit and durable.

### 1. Blank prompt is not an acceptable contract signal

- Blank or whitespace-only `prompt` must not be the architectural meaning of control-only resume or control-only fork.
- This stack must not rely on transitional blank-prompt compatibility as part of the target design.

### 2. Control-only resume is a Substrate-owned requirement

- Based on the current live Substrate runtime, the real pressure is control-only host recovery for `reattach`.
- Prompt-bearing `turn` already covers normal follow-up on both host and Linux world-member paths.
- No current live Substrate path proves a durable need for promptless backend-native fork.
- The current evidence does not prove that control-only resume must be promoted into UAA as a shared backend contract.
- The current evidence does suggest that the hidden owner-helper seam should be decomposed into explicit Substrate-owned responsibilities rather than remaining one semantically blurry runtime concept.

### 3. Promptless fork is not a forward dependency

- Substrate must not make promptless `agent_api.session.fork.v1` part of the required floor for the next slices.
- Public `fork` remains a Substrate control action that allocates a successor durable orchestration session and lineage.
- If backend-native fork-without-turn is ever needed later, it must land as an explicit, documented, capability-gated operation rather than as “blank prompt fork.”

### 4. Public world-scoped root start is host-rooted only

- The only aligned meaning of a future public `substrate agent start --scope world ...` in this stack is:
  - create a host-rooted orchestration session,
  - launch a world worker/member under that session,
  - defer host execution-client attach until real host work exists.
- Standalone world-root continuity is not part of this SOW chain and must not remain an open branch inside `30` or `31`.

## Current Repo Truth

### Substrate truth

What is already true in live code:

1. Prompt-bearing host execution and prompt-bearing Linux world-member execution fulfill through gateway-mediated bridges in:
   - [crates/shell/src/execution/prompt_fulfillment.rs](../crates/shell/src/execution/prompt_fulfillment.rs)
   - [crates/world-service/src/prompt_fulfillment.rs](../crates/world-service/src/prompt_fulfillment.rs)
2. Public `turn` is still prompt-bearing follow-up on the same durable orchestration session.
3. Public `reattach` is intended to restore attached ownership without submitting a prompt.
4. Detached-world follow-up remains fail-closed until host ownership is restored.

What is still mismatched:

1. Hidden owner-helper `Resume`, `ResumeOneTurn`, and `Fork` still share one resume-extension startup path.
2. `NoPromptRecovery` still becomes `prompt: ""` before the UAA request is built.
3. Live `fork` does not currently prove a real backend-native fork requirement; it looks more like successor durable-session allocation plus backend-native resume continuity.
4. `HOST_ORCHESTRATOR_INTENDED_BEHAVIOR_TRUTH.md` should be synchronized so it explicitly reflects the shipped public `fork` surface rather than omitting it from the frozen public contract.
5. The hidden owner-helper recovery path is the place where control-only pressure currently exists, and that pressure originates from Substrate-owned orchestration/runtime shaping rather than from a proven cross-backend UAA contract gap.

### Unified Agent API truth

What is already true in the evaluated UAA branch:

1. Blank prompts are accepted when `agent_api.session.resume.v1` or `agent_api.session.fork.v1` is present.
2. Codex promptless resume is currently implemented as a control-only resume path.
3. Codex promptless fork is currently implemented as a control-only fork path that returns a handle without starting a turn.
4. Claude also currently supports blank-prompt resume/fork behavior through `no_prompt()`-style CLI construction.

What remains mismatched:

1. The normative run protocol still requires non-empty prompt validation before spawn.
2. The normative extensions spec still describes `resume` and `fork` as prompt-bearing operations.
3. The backend-specific Claude and Codex contract docs still pin non-empty prompt semantics.
