# Implementation Plan: World-First Shell Execution Model

## Recommended Implementation Path – Persistent World PTY Session (Option A)

* **Choose Persistent PTY-Backed Shell:** Implement the REPL using a long-lived shell process inside the isolated “world” environment, rather than per-command execution. This means Substrate will spawn and maintain a persistent interactive shell (e.g. a Bash session) within the world, and feed each user-entered command into that session[\[1\]](file://file_00000000a948722f938ff47e518d0244#:~:text=Option%20A%20,input%20line%20within%20that%20session)[\[2\]](file://file_00000000a948722f938ff47e518d0244#:~:text=Decision%20,backed%20world%20session). This was selected over a “virtual state” approach because it **matches user expectations** – builtins like cd or export naturally persist and behave correctly as in a normal shell, with fewer special cases or surprises[\[3\]](file://file_00000000a948722f938ff47e518d0244#:~:text=Tradeoffs%20,exit%20codes%2C%20and%20fs%20diff). By contrast, simulating shell semantics in the host process (Option B) would require reimplementing environment and directory tracking (with tricky edge cases in quoting and parsing) and could still get out of sync with the real world state[\[4\]](file://file_00000000a948722f938ff47e518d0244#:~:text=,state%20diverges%20from%20the%20world). The persistent world shell ensures **full fidelity**: cd, pwd, export, unset and even complex shell features (aliases, functions, job control) work as expected within the world[\[5\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,filesystem%20view%20the%20user%20sees)[\[6\]](file://file_00000000a948722f938ff47e518d0244#:~:text=Tradeoffs%20,command%20trace%20fidelity).

* **Justification:** The current REPL has a split personality – most commands run in the world’s overlay filesystem, but a handful of builtins (cd/pwd/export/unset) execute on the host, affecting host cwd and env. This leads to inconsistent behavior (e.g. a directory created in the world exists there, but a subsequent cd fails because the host can’t see it)[\[7\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,not%20require%20users%20to%20know)[\[8\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=request%2Fresponse%20paths.%20,the%20world%2C%20but%20it%20is). By moving to a persistent in-world shell, **all commands run in one context**, eliminating these “mixed context” failures. The REPL will *behave like a normal shell inside the world by default*, aligning with user intuition that when the world is enabled, they are essentially operating inside an isolated environment[\[9\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,filesystem%20view%20the%20user%20sees)[\[10\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=executed%20by%20the%20host%20process,path%20on%20the%20host%20filesystem). This design dramatically simplifies mental models: users no longer need to guess which commands are being executed on the host vs in the world.

* **Operator Experience:** With Option A, the Substrate shell will automatically start a world session on launch (for Linux, and Mac via its Lima VM) and execute **all un-prefixed commands inside that world session**[\[11\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,session%20as%20part%20of%20cleanup)[\[12\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,be%20recognized%20in%20this%20mode). The host environment is still accessible, but only via an explicit prefix (see “Host Escape Hatch” below). This world-first default preserves isolation guarantees while still allowing access to host tools when explicitly needed, making the default workflow safer and more predictable[\[13\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=stateful%20builtins%20%28,only)[\[14\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,first%20and%20predictable). Importantly, each command entered will still produce its own trace/span and output, but under the hood they are evaluated by the persistent shell process.

* **No Per-Command VM Spin-Up:** This approach reuses a single world instantiation for the whole REPL session, avoiding repeated world setup/teardown for each command. That keeps performance reasonable and allows state to persist across commands (e.g. changing directories once affects subsequent commands, environment variable exports remain in effect, etc.), which was a key goal[\[15\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=Goals%20,only)[\[16\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,first%20semantics%20when).

In summary, **Persistent World PTY (Option A) is the recommended path** because it provides correct shell semantics (directory changes, env vars, etc.) with minimal surprises[\[3\]](file://file_00000000a948722f938ff47e518d0244#:~:text=Tradeoffs%20,exit%20codes%2C%20and%20fs%20diff). Although it introduces some implementation complexity (managing a live shell process), it aligns the REPL behavior with the isolation model and user expectations of a “world shell”[\[17\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,shape). The alternate approach of simulating shell state in the host (Option B) was deemed too fragile and inconsistent, as it risks divergence between Substrate’s internal state and the real world state and would require duplicating complex shell logic[\[4\]](file://file_00000000a948722f938ff47e518d0244#:~:text=,state%20diverges%20from%20the%20world). The ADR decision DR-02 confirms that a persistent PTY-backed session was chosen as the default world-first strategy[\[18\]](file://file_00000000a948722f938ff47e518d0244#:~:text=Option%20A%20,input%20line%20within%20that%20session)[\[19\]](file://file_00000000a948722f938ff47e518d0244#:~:text=Decision%20,backed%20world%20session) for these reasons.

## Key Risks and Edge Cases to Address

Implementing a persistent in-world shell session introduces several challenges and edge cases that must be handled carefully:

* **Command Boundary & Exit Codes:** In a long-lived shell process, we need a **robust way to delineate commands** and retrieve each command’s exit status without terminating the shell[\[20\]](file://file_00000000a948722f938ff47e518d0244#:~:text=,command%20fs%20diff%20capture). Unlike the per-command model (where the process exit code is directly available), a persistent shell will continuously run. **Solution:** Introduce a command-boundary protocol – for example, configure the shell to emit a unique marker (possibly via PROMPT\_COMMAND or a shell wrapper) after each command, encoding the exit code. The Substrate client can watch the shell’s output for this marker to determine when a command has finished and capture its exit status. This marker approach must be deterministic and unlikely to appear in normal command output (e.g. a special control sequence or token). We’ll implement and test that the client reliably detects end-of-command even for commands that produce large output or for interactive programs. This addresses the **“robust command-boundary protocol”** concern noted for Option A[\[20\]](file://file_00000000a948722f938ff47e518d0244#:~:text=,command%20fs%20diff%20capture).

* **PTY Session “Drift” or Desync:** With a persistent PTY, there’s a risk the shell’s state could drift out of sync with what Substrate expects. For example, if the user manually backgrounded processes or sent special control characters, would our front-end handle it? We must ensure that **signals and job control** are handled or explicitly documented. The current design will primarily target foreground command execution. We will capture common cases like Ctrl+C (SIGINT) and forward them to the world shell process (this is already part of the WS protocol) to allow users to cancel long-running commands. Suspending (Ctrl+Z) and backgrounding jobs are advanced cases – likely we will document that job control in the Substrate REPL is limited, or we may disable interactive job control in the shell for now (e.g. by starting bash with monitor mode off). **Terminal raw mode** considerations (for full-screen programs like editors) are already handled via the PTY streaming; we need to ensure those continue working in a persistent session. This means verifying that when a program like vim is run inside the world shell, the PTY handles switching to raw mode and back properly on exit. These interactive program and job-control nuances are called out as a complexity of Option A[\[21\]](file://file_00000000a948722f938ff47e518d0244#:~:text=,command%20fs%20diff%20capture) and will require careful testing.

* **Filesystem State & fs\_diff Capture:** In the current per-command model, each isolated command can produce a filesystem diff (FsDiff) that is recorded in the trace for auditing. With a persistent session, the overlay filesystem continues across commands, so we need a strategy to **capture per-command fs\_diff** without destroying the session[\[20\]](file://file_00000000a948722f938ff47e518d0244#:~:text=,command%20fs%20diff%20capture). We can approach this by computing a diff of the overlay mount before and after each command execution. For example, after a command finishes (detected via the marker), the Substrate client or world-agent can invoke a lightweight diff operation (perhaps using our OverlayFs utility) to snapshot changes made by that command. This diff can then be attached to that command’s span. We must also consider performance: frequent diffing could be expensive for large projects, so we might optimize by tracking touched paths or using copy-on-write info if available. Additionally, long-running sessions that make many file changes might accumulate state that complicates diffing. We will document that diffs in PTY sessions are “best effort” – e.g. if a command is still running or if the user has background processes modifying files, the diff may not capture everything until the command boundary. This aligns with known **PTY diff caveats**[\[22\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/BACKLOG.md#L34-L42). Our implementation will aim for parity: each REPL command’s trace entry should include fs\_diff just like non-PTY commands[\[23\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/BACKLOG.md#L34-L41), but we’ll note any limitations (like partial diffs for long interactive sessions)[\[24\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/BACKLOG.md#L34-L38).

* **Environment & CWD Synchronization:** In a persistent shell, the **shell’s own environment and working directory become the source of truth** for subsequent commands. We must ensure that the initial environment passed to the world shell is set up correctly (mirroring any necessary host env vars, PATH, etc., minus anything unsafe). Once running, commands like export VAR=... or cd /path will update the world shell’s state. There is a potential edge case if Substrate’s host process also tracks some state (for example, the current directory on the host). In the new model, **the host process should no longer change its own cwd or env for builtins** – those operations happen in-world. Indeed, the ADR notes that previously Substrate would mutate host cwd for cd as a “lightweight builtin”[\[25\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,path%20on%20the%20host%20filesystem); that will be removed. We will treat the world shell’s state as authoritative. One risk is if some internal operations still rely on host state (e.g. if Substrate needs to read a file from the project directory, it must use the world’s view). We’ll audit any such cases. The design also needs to consider that certain host-derived values (like $HOME or config paths) might differ in the world. Our world environment initialization will carefully propagate essential variables (possibly via the broker’s policy). For instance, $HOME in the world might be an isolated home; this is acceptable as long as tools inside the world use that. We will double-check that any environment *not* propagated doesn’t break common tooling (e.g., no XDG\_\* surprises). In summary, **internal shell state is fully handled in-world**, and we avoid mixing host values except where explicitly intended (such as a controlled set of forwarded env vars).

* **Policy Snapshot Consistency:** The broker provides policy inputs (like allowed network domains, isolation flags) at world session start. In a persistent session model, all commands in that session will use the **same policy snapshot** captured at creation time[\[26\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/PHASE_4_5_ALWAYS_WORLD_IMPLEMENTATION_PLAN.md#L46-L54). We need to ensure this remains consistent and is recorded for traceability. For example, the set of allowed egress domains (for network isolation) is determined once when the world is created and applied via netfilter rules[\[27\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/ALWAYS_IN_WORLD_PTY_EXECUTION_PLAN.md#L80-L89)[\[28\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/ALWAYS_IN_WORLD_PTY_EXECUTION_PLAN.md#L94-L101). If the policy changes mid-session (which is uncommon unless an admin modifies the policy file during the session), the running world won’t automatically get those updates. This is acceptable as a first iteration – we document that policy (like allowed\_domains) is evaluated when the session starts. If needed in the future, we could add a mechanism to update the running world’s netfilter rules or require users to restart the shell to pick up policy changes. The key is that **each command’s execution should log which policy snapshot it was under**, to avoid ambiguity. We’ll include metadata in each command span (like a policy version or the world session ID) to indicate that all commands in a session share the same isolation parameters. This ensures trace auditability that the same world (ID) and same sandbox rules were in effect[\[29\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/BACKLOG.md#L38-L44). There’s also a **consistency check between broker and world-agent**: when the world session is launched, the broker’s decision (to enable world, what isolation mode, etc.) must be respected by the agent. We will test scenarios like: world required by policy but world-agent not running – Substrate should *not* silently fall back to host. Instead it must warn or error (see Security below).

* **Security & Host Escape Risks:** A primary motivation for world-first is security – preventing unintentional host execution. We must scrutinize any code paths where if the world is unavailable or fails, Substrate might fall back to running a command on the host. Our new design will enforce **fail-closed behavior** when isolation is expected. If the user’s config or policy sets world\_fs.require\_world=true, and the world cannot be started, the shell should refuse to run commands on the host (except perhaps allowing a :host if the operator explicitly chooses)[\[30\]\[31\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,and%20exit%20status). We will implement *high-signal diagnostics* in this case: e.g. on startup, if world init fails and require\_world is true, print an error and exit (non-zero) rather than dropping into an unisolated shell. If require\_world is false (the default), we may allow a fallback to host with a **clear one-time warning** that the world is unavailable and we are in host mode. This warning already exists in current code (e.g. “world isolation unavailable (observe-only)…”[\[32\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/PHASE_4_5_ALWAYS_WORLD_IMPLEMENTATION_PLAN.md#L60-L64)); we’ll ensure it only appears once and is prominent. Additionally, we guard the new :host prefix itself: it should never be honored in non-interactive contexts or by automated scripts, and even in the REPL it will be gated (see below) to prevent accidental use. This gating (requiring a flag to enable :host) is specifically chosen to **prevent automation or unaware users from bypassing isolation**[\[33\]](file://file_00000000a948722f938ff47e518d0244#:~:text=Option%20A%20,c%60%20or%20CI%2Fautomation%20modes)[\[34\]](file://file_00000000a948722f938ff47e518d0244#:~:text=,host%20execution%20during%20interactive%20use). In testing, we will verify that :host commands do nothing if the feature isn’t enabled (likely we will print a message like “host escape is not enabled” and ignore the input).

* **Interactive Quoting and Input Handling:** Feeding commands into an interactive shell can introduce quoting/escaping issues if not done carefully. We’ll need to ensure that when Substrate sends a line of input over the WebSocket to the world shell, it doesn’t get munged. Likely, we will send the exact text as entered (the shell will interpret it normally). We should avoid wrapping it in an extra bash \-c "" in this mode (unlike the non-PTY path) because the persistent shell will parse it. This actually reduces quoting issues: the user’s text is handled by the shell’s parser exactly as they intended. We just need to be cautious with our end-of-line marker protocol (it should not interfere or be exposed if, say, a command tries to read from stdin or produce similar output).

In summary, the major risks (command boundary detection, PTY state management, fs diff correctness, and security fallbacks) are known and will be mitigated with careful protocol design and comprehensive testing. We will create dedicated tests for each edge case (see Test Plan) to ensure the new model doesn’t introduce regressions in behavior or security.

## Breakdown of Required Code Changes

Achieving the world-first REPL will involve updates across multiple parts of the codebase:

* **Shell REPL Loop & Command Routing:** The substrate-shell crate’s interactive loop needs substantial adjustment. Currently, it special-cases builtins like cd and handles command dispatch via either local exec or world-agent per command. We will **remove the host-builtins fast path** for cd/pwd/export/unset when world mode is active[\[25\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,path%20on%20the%20host%20filesystem). Instead, on each input line, the shell will check for the :host prefix (if present and allowed) or otherwise route the command to the world session. In practice, this means:

* Initializing the world session at shell startup (already partly done via ensure\_session() in run\_shell[\[35\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/PHASE_4_5_ALWAYS_WORLD_IMPLEMENTATION_PLAN.md#L30-L39)[\[26\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/PHASE_4_5_ALWAYS_WORLD_IMPLEMENTATION_PLAN.md#L46-L54), which returns a world handle/ID). We will likely extend the Shell’s state to hold a reference to the active world session or a client connection to the world-agent’s streaming endpoint.

* For **unprefixed commands**: instead of directly calling backend.exec() for each command, the shell will send the command over the persistent connection to the world shell. The code in execute\_command() will be refactored: if a persistent world session is live, it writes the input to that session (and then waits for the result/marker). The logic that previously handled printing command output and exit codes will remain, but now it reads from the session’s output stream.

* For **builtins** like cd: we will no longer call std::env::set\_current\_dir on the host. They’ll just be treated as normal text to the world shell. The effect (changing directory or env inside world) happens within the persistent shell.

* The REPL prompt should update to reflect the current world working directory. Since the host is no longer tracking cwd, we can’t use host cwd for the prompt. Instead, after each command (or before displaying the next prompt), we could query the world for its cwd (for example, by running a quick pwd or by parsing our marker if it includes the cwd). We might set the shell prompt to always show something like (world:${PWD})\>. One efficient approach: leverage the shell’s ability to set the prompt string. We can configure the world shell’s PS1 to a zero-length string (so it doesn’t output its own prompt), and let the Substrate process render the prompt on the host side using the world session’s state that it tracks. This requires tracking the cwd: we will update it whenever a cd command is executed (we can intercept cd by pattern or possibly have the marker output include the new cwd).

* **Executing Host Commands:** We will implement recognition of the :host prefix in the REPL input parser. If a line begins with :host, Substrate will strip that and execute the remainder of the command *on the host* (basically the old behavior). This will use the existing local execution path (with proper sh \-c handling for complex commands). We’ll also likely treat :host cd and other builtins by performing them on the host (since in that scenario the user explicitly wants to affect host state)[\[36\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,NOT%20execute%20on%20the%20host)[\[37\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,accidental%20or%20programmatic%20host%20execution). These code paths need to be clearly separate to avoid any confusion with the world session.

World-Agent and World Backend (All Platforms, Always via world-agent)

Substrate always executes through world-agent, regardless of platform (Linux/macOS/Windows). There is no in-process/local-backend bypass path for command execution or PTY. This ensures one consistent execution surface, one consistent policy enforcement boundary, and one consistent observability/tracing pipeline.

Implications / shape:

Single execution contract:
All execution modes route to world-agent:

Non-PTY one-shot execution: POST /v1/execute

PTY streaming execution: GET /v1/stream (WebSocket JSON PTY protocol)

World creation & session lifecycle are agent-owned:
The shell does not create/enter namespaces directly. Instead:

The shell requests execution with the appropriate world/session identifiers and config payloads.

world-agent is responsible for session creation, overlay mount setup, Landlock / allowlists, cgroup attachment, and (when enabled) network isolation.

Persistent REPL shell session = persistent /v1/stream session:
For Option A, the REPL starts a single long-lived PTY-backed shell by opening one /v1/stream connection and issuing a start frame whose cmd launches an interactive shell (e.g., bash), inside the world.

The REPL then sends each user-entered line as stdin frames to that same session.

Output arrives as stdout frames.

Substrate derives per-command completion + exit codes via a command-boundary mechanism (markers) without terminating the PTY session.

Policy snapshot as session configuration (drift handled explicitly):
The start frame includes cwd, env, policy_snapshot, and any world/session mode fields. This snapshot configures the world session. If policy/cwd changes require a different snapshot, the design must choose (and enforce) one:

restart the PTY session with a new start snapshot, or

extend the PTY protocol to support a reconfigure snapshot message.

Parity + correctness:
Because Linux also routes through world-agent, the Linux path must use the same env normalization and request-building helpers as other platforms (eliminating the current “Linux PTY uses std::env::vars()” inconsistency).


  * **Session Lifecycle:** The world-agent may need to maintain the session context so that subsequent /v1/execute calls don’t conflict or try to reuse the same namespace. However, because we are using one shell for the REPL, we actually won’t be calling /v1/execute at all for REPL commands anymore – it’s all within the /v1/stream. So one design point: do we allow multiple concurrent shell sessions? Likely not in a single REPL. The agent should treat each WebSocket /v1/stream connection independently (it already does), and each such connection can have its own world session. We will use the reuse\_session: true flag to ensure the same session overlay is reused if possible[\[43\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/ALWAYS_IN_WORLD_PTY_EXECUTION_PLAN.md#L132-L135), but since only one REPL is active per Substrate invocation, we effectively have one world per shell.

  * The world-agent protocol itself might not require new message types, since we can use the stdout stream for markers. We will document that the **command boundary marker is handled on the client side**; the server (world-agent) just shuttles bytes. If we find it unreliable to parse in the client, we could consider enhancing the protocol (e.g. the agent could be taught to recognize our marker and send a special JSON message with the exit code). However, that would involve modifying the agent’s code to parse the child process output, which we may avoid for now. Simpler is better: the agent remains dumb pipe, and the client parser handles markers.

* **Broker / Policy Changes:** The broker (policy engine) primarily influences which world is created (and with what settings). For example, it supplies allowed\_domains and flags like isolate\_network that get built into the WorldSpec[\[26\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/PHASE_4_5_ALWAYS_WORLD_IMPLEMENTATION_PLAN.md#L46-L54). We don’t need to change the broker logic, but we should ensure that:

* We call the broker at shell startup to get the latest policy snapshot (already done via substrate\_broker::allowed\_domains() etc. when building WorldSpec[\[26\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/PHASE_4_5_ALWAYS_WORLD_IMPLEMENTATION_PLAN.md#L46-L54)).

* If policy includes something like world\_fs\_mode \= read\_only, the world session honors it (the SessionWorld logic already mounts overlay read-only accordingly).

* If always\_isolate is set, that will force use of overlay for every command (which is fine, since our persistent shell will by default be running on the overlay filesystem anyway).

* The **policy snapshot ID or content** could be logged in the session for traceability. Not strictly required, but helpful. Potentially log an info line like “World session started with policy X (network isolated: true, allowed\_domains: \[...\])”.

* No changes needed to the broker interface, just usage.

* **Telemetry & Tracing:** We must update the tracing instrumentation to reflect the new execution model:

* **World ID in Spans:** Ensure that each command’s trace span includes the world\_id of the session if one is in use. Currently, interactive commands logged world\_id: null because they bypassed the agent path[\[29\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/BACKLOG.md#L38-L44). Now that everything truly goes through the world (either via agent or in-process), we can populate this. When the session is created, we have a world ID (like wld\_xxx); we will store it and when starting a new command span, include world\_id \= that\_id in the metadata[\[29\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/BACKLOG.md#L38-L44). We’ll adjust the trace logging in shell (likely in span.finish() calls or span fields) accordingly.

* **Execution Origin:** We add an explicit field or tag for execution\_origin as noted in the ADR[\[31\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,and%20exit%20status). For each command, mark it as world or host. By default in the REPL, this will be world for all unprefixed commands (since they run in the world session) and host for any :host command. This way, analysis of trace.jsonl can clearly differentiate isolation context[\[31\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,and%20exit%20status). We might implement this by setting a tag on the span when we know it’s going to the world vs host.

* **CWD tracking:** It could be useful to log the working directory for each command as seen in the world. In the current design, because host and world could differ, it was tricky. With world-first, we can likely log the world cwd. One approach: after a command completes, query the world’s cwd (e.g., run pwd hiddenly or use an ioctl if available). But this might be too invasive per command. As an alternative, since we are feeding the commands, we know the cwd initially and we track internally when cd commands are executed. For trace purposes, just logging the initial cwd of the world session might suffice (the project root), and relative paths of commands can be inferred. This is a nice-to-have; ensuring correctness of origin and world\_id is higher priority.

* **Logging Command Boundaries:** We will also integrate with our existing logging of command start and completion. When using the streaming path, we likely have to emit a “command started” event when we send it to the world, and then “command completed” when the marker arrives with an exit code. We need to ensure that our span lifecycle (start \-\> finish) wraps exactly one user command, even though the underlying shell process stays alive. This might require changes in how we manage spans: possibly keeping a span open while waiting on the marker, then closing it and creating a new one for the next command.

* **Performance metrics:** This change shouldn’t heavily impact performance, but we should monitor if any new latency is introduced (e.g., computing fs\_diff each time, or waiting on markers). Our tracing could include timing for each command as usual.

* **CLI & Configuration:** We will add a new CLI flag or config to control the host escape hatch availability (and possibly any legacy fallback, though we decided not to include a long-term legacy mode):

* For :host gating, introduce something like \--allow-host or an environment variable (SUBSTRATE\_HOST\_ESCAPE=1) that a user must enable if they want to use :host in the REPL. By default, without this flag, typing :host ls would result in an error or no-op (so that automation can’t accidentally run it)[\[36\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,NOT%20execute%20on%20the%20host)[\[44\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,it%20must%20be%20gated%20to). We’ll implement this check at REPL startup (and possibly show in the welcome message whether host escape is enabled or not). The decision was to **require explicit opt-in** for :host[\[33\]](file://file_00000000a948722f938ff47e518d0244#:~:text=Option%20A%20,c%60%20or%20CI%2Fautomation%20modes)[\[34\]](file://file_00000000a948722f938ff47e518d0244#:~:text=,host%20execution%20during%20interactive%20use), so we abide by that.

* The \--no-world flag and SUBSTRATE\_WORLD=disabled env will still function: if the user explicitly disables world isolation, Substrate should bypass all the above and run as a pure host shell (essentially the old behavior). We need to ensure our routing respects this: if no world, then :host prefix is moot (everything is host anyway), and we must use the old builtin handling. However, since we’re unifying semantics, even in no-world mode we might consider removing special-cased builtins (they’re unnecessary if everything is on host). But to minimize changes, we can leave the host-only path as is when world is disabled.

* \-c/--command handling will be updated as described: if world is enabled, we do *not* do host builtin shortcuts[\[45\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,the%20operator%20expects%20world%20semantics). This means in the code path for one-shot commands (likely in main or early argument handling where \-c is processed), we should force the use of the world backend for the entire command string. Concretely, if \--command "cd /foo && ls" is invoked and world is on, we will start a world (if not already), then run the command via backend.exec or an equivalent that interprets it in a shell. The ADR says :host is never recognized in \-c[\[46\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,the%20operator%20expects%20world%20semantics)[\[12\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,be%20recognized%20in%20this%20mode), so we don’t need to parse for that. And builtins like cd will only affect the subshell running that single command (which is fine – it won’t persist after, consistent with normal shell behavior in a bash \-c context). We’ll add tests to ensure that, for example, substrate \-c "cd someDir; pwd" returns the path of *someDir* if world is enabled (meaning it actually changed directory within the sub-shell), whereas previously it might have printed the host’s original directory or even failed if someDir was only in overlay.

In summary, changes span the Shell (prompt, input parsing, dispatch), the World Agent (ensuring a persistent session and proper isolation for the shell process), and Telemetry. The broker and core policy logic remain the same, but we integrate their output at session start. We anticipate most code changes to be in crates/shell (especially modules under execution/routing for commands) and crates/world-agent (PTY handling), plus minor adjustments in crates/world (session management if needed) and crates/trace (for new fields like execution\_origin).

## World-Agent Protocol & Session Management Adjustments

Introducing a persistent session necessitates some refinements to the world-agent communication protocol, especially for cross-platform support (Linux vs macOS):

* **Persistent Session via /v1/stream:** We will leverage the existing WebSocket streaming endpoint as the transport for the persistent shell. The protocol messages remain as they are (startup, stdout, stdin, etc.)[\[40\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/ALWAYS_IN_WORLD_PTY_EXECUTION_PLAN.md#L38-L46)[\[47\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/ALWAYS_IN_WORLD_PTY_EXECUTION_PLAN.md#L44-L46). The key change is the **command sent on start:** instead of running the user’s command, we start an interactive shell. For example, the client will send {"type":"start","cmd":"bash \-i", "cwd": "\<proj-dir\>", "env": {…}, ...} (plus initial terminal size, span\_id, etc.)[\[40\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/ALWAYS_IN_WORLD_PTY_EXECUTION_PLAN.md#L38-L46). We might include additional commands to configure the shell: e.g., bash \-i with certain environment variables set (PS1, markers). The world-agent doesn’t need a new endpoint for this; it will execute that bash inside the world’s namespaces and keep the PTY open. From the agent’s perspective, it’s just another long-running process. We will ensure the agent’s handler for /v1/stream attaches the process to the correct cgroup and netns of the session[\[39\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/ALWAYS_IN_WORLD_PTY_EXECUTION_PLAN.md#L126-L134)[\[48\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/ALWAYS_IN_WORLD_PTY_EXECUTION_PLAN.md#L132-L140) (as noted earlier, Patch D). Once running, the agent will forward all output from bash to the client as stdout messages, and forward any client keystrokes (stdin messages) to the bash process.

* **Command Boundary Markers:** As discussed, the protocol currently doesn’t delineate separate commands within one stream – it’s up to us. We will implement markers in the **shell’s output**. This likely does not require changes to the agent, only to how we invoke bash or how we parse output. One idea: set PROMPT\_COMMAND='echo \-en "\\u0099SUBDONE:$?:"' (where \\u0099 is a rarely used control char as a marker start, for example). When bash finishes a command and is about to show a prompt, it will output our sequence containing the last exit code. The client can intercept this specific byte pattern and not forward it to the user, instead using it to conclude the span. This approach keeps the agent unaware of markers. We must verify that on macOS (Lima) and Windows (WSL) the control sequence passes correctly. **Alternative:** If we wanted a cleaner solution, we could extend the WS protocol to have a special message type like {"type":"cmd\_end","exit":N}, but since the agent would have to generate that by parsing the child process status, it complicates the agent. We’ll stick with client-side parsing for now as it’s decoupled and easier to iterate on.

* **Session Reuse & Snapshot Refresh:** The world-agent already supports an ExecRequest.reuse\_session flag and the concept of a session handle (though find\_existing is not implemented yet)[\[49\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/crates/world/src/session.rs#L26-L34)[\[50\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/crates/world/src/session.rs#L61-L69). For persistent REPL, reuse is naturally true: we want one session for the whole REPL. If a user exits and restarts Substrate, they’ll get a new session (we are not implementing “resume previous session” at this time, though it’s noted as an idea in backlog[\[51\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/BACKLOG.md#L73-L81)). We do not plan to allow *reconnecting* to an existing world shell in this iteration – that would require the shell to remain running after Substrate exits, which isn’t in scope. Each REPL invocation manages its own world lifecycle (create on start, tear down on exit). The world-agent might not need any special tracking for this beyond what it does now (clean up the namespaces and mounts when the process exits).

* **Closing Sessions:** Ensure that when the user exits the REPL (via exit or Ctrl+D), we gracefully terminate the world shell process and cleanup. The client can simply drop the WebSocket connection (or send a :q to the shell). The agent should detect the socket close and kill the child process if it’s still running. This behavior exists (WS on\_upgrade with a future that ends when socket closes). We will test that no world processes or mounts leak after exit.

* **Reconfigurable Snapshots / Allowed Domains:** If we consider advanced use-cases, one might want to adjust network policy on the fly. While not in scope for now (policy changes require restart), our design leaves room: in theory, we could add a control message to request the world-agent to update allowed domains and refresh netfilter rules. Similarly, if one wanted to toggle fs\_mode mid-session (say from read-only to writable), that would be complex (fuse-overlay cannot easily flip to writable without remount)[\[52\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/crates/world/src/session.rs#L317-L325)[\[53\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/crates/world/src/session.rs#L334-L343). So we will **not attempt dynamic reconfiguration** of the world once started; any such changes require ending the session and starting a new one. We’ll document this as a limitation: e.g., “Changes to isolation settings or policy will apply on the next session start.”

* **Extensibility to macOS (Lima):** The same WS protocol and approach will apply to Lima’s world-agent (the agent runs inside the Lima VM, which is Linux). We need to ensure path translation is handled: the cwd we send in the start request on macOS must likely be translated to the VM’s mount point path. The substrate code already has logic for converting host paths to WSL or Lima paths (e.g., using /mnt/c/... for Windows, and whatever Lima uses)[\[54\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/PHASE_5_ALWAYS_WORLD_WINDOWS_PLAN.md#L38-L46)[\[55\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/PHASE_5_ALWAYS_WORLD_WINDOWS_PLAN.md#L40-L45). We will reuse those utilities so that if the user’s project directory on macOS is /Users/alice/proj, and in Lima it’s mounted under /mnt/lima/substrate/proj, we send the Lima path in the cwd field of the start message. Similarly for any environment variables like PWD or others, though likely just using the proper cwd is enough for the shell to know where it is.

* **Terminal Mode Mismatch:** One known issue has been differences in terminal handling across platforms. For example, Windows uses ConPTY and might not support some ioctls, etc. However, since Windows interactive world support is a separate future (explicitly out-of-scope in ADR)[\[56\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,specific%20design), we will focus on Linux/macOS where the PTY model is well-supported. We will verify that our approach (e.g., using control characters for markers) doesn’t interfere with terminal encoding. Lima’s vsock-proxy and our WS channel should pass binary data transparently. We must be cautious that the marker sequence we choose doesn’t get munged by any layer (for instance, avoid newline alone as a marker, since that’s common).

* **Logging and Auditing on Agent:** The world-agent will log the start of the session (already likely logs when a command is executed). We might enhance logging to note that a persistent shell was launched for session X. Each command won’t individually hit the agent’s HTTP /v1/execute, so the agent’s own logs will just show the PTY session. That’s fine. For debugging, it may be harder to see what commands were run in the world from the agent side (since it’s just a stream). But we have the trace on the client side for that. If needed, we could have the agent echo commands to its log; but that might duplicate client info. We’ll likely not add that to avoid noise.

In summary, **no major changes to the wire protocol** are needed; we primarily reuse the existing streaming mechanism but repurpose it to run an interactive shell. The critical addition is a client-side command demarcation scheme (markers) and ensuring the world-agent properly sets up the shell in the isolated environment. The design is inherently extensible: when we tackle Windows, we’ll implement a similar concept using ConPTY (spawning a Linux shell in WSL via a named pipe/ConPTY bridge – Phase 5 work, outside this plan). The Linux/macOS implementation will serve as a blueprint, and the protocol (WS with start/stdin/stdout) remains consistent across platforms.

It should print the host’s current directory (likely the project dir on host, since host process hasn’t chdir’d since startup in this new model – which will be the project root unless we decide to change host cwd management). If the world wasn’t actually used, it would not find it. This verifies that one-shot commands use the world properly.

* substrate \-c "cd subdir && touch newfile": after running this, since it’s non-interactive, the cd shouldn’t persist (it only affects that sub-shell). But we expect newfile to be created inside subdir in the world. We can check that by running another command or from host (looking into the world overlay). Essentially, ensure that the command ran with world’s view of filesystem (if subdir didn’t exist on host but did in world, the touch should succeed and create the file in overlay).

* Ensure that the exit code of the substrate \-c reflects the world command’s exit. e.g., substrate \-c "false" should return non-zero. We document that :host cannot be used in \-c. A test should validate that it doesn’t somehow escape to host. (If we want, we might even have substrate detect and refuse if the command starts with :host in non-interactive mode, but simply doing nothing special already means it won’t work, which is safe).

* **Failure and Fallback Tests:**

* Simulate world-agent not running or world unsupported (e.g., on a kernel without user namespaces). Ensure that when starting the shell, Substrate prints the appropriate warning (“world isolation unavailable”) and either continues in host mode (if policy allows) or exits (if require\_world). If continuing in host mode, verify that builtins then operate on host (since we fell back entirely).

* Toggle the require\_world policy and repeat: with require\_world, the shell should abort launching with a clear error if it cannot initiate the world. This can be tested by e.g. temporarily moving the world-agent binary or causing an intentional failure.

* **Concurrency/Telemetry Tests:**

* Start a REPL, run a series of commands quickly to ensure the event loop can handle back-to-back without losing the marker synchronization (this might be tested via an automated script driving the REPL).

* Check the trace.jsonl output after a session: ensure that each command\_complete event has world\_id populated (not null)[\[29\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/BACKLOG.md#L38-L44), and that execution\_origin is “world” for all but any host-prefixed commands, which should show “host”. Also verify cwd if we log it, and that fs\_diff appear for commands that made changes (the diff might not list everything changed in entire session, but at least per command changes if we implemented it).

* If possible, test the size of fs\_diff on a large change (like echo foo \> bigfile of 10MB) to see performance.

* **Manual Validation:** In addition to automated tests, a thorough manual QA will be done:

* Using the interactive shell in a realistic development scenario: e.g., create a Python virtualenv in the world, install a package, verify isolation (the package installed doesn’t affect host).

This test plan ensures we cover correctness of new behavior, backward compatibility in edge cases, security (no unintended escapes), and that the developer experience remains smooth. We will iterate on the implementation guided by these tests, fixing any failing scenario.

## Handling Rules for :host and Non-Interactive Commands

**Interactive Host Escape (:host):** We introduce the :host prefix as the *only* way to run commands on the host from an interactive session. By design, this is an explicit opt-in mechanism: \- In interactive mode, when the operator has enabled host escapes, any input line starting with :host will be executed on the host system instead of in the world[\[14\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,first%20and%20predictable)[\[36\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,NOT%20execute%20on%20the%20host). We parse out the prefix and run the rest of the line via the standard host shell (respecting the host $PATH, etc.). For example, :host docker ps would run the host’s docker command, and :host cd /tmp would change the directory of the *host* Substrate process (which in this mode we will allow, affecting subsequent host commands’ context but **not** the world’s cwd). We will document that :host cd affects only the hidden host working dir (since the world’s cwd remains separate). \- We explicitly **do not allow** any form of :host in non-interactive contexts. In a substrate \-c ... invocation or any script/CI usage, :host will simply not be recognized[\[46\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,the%20operator%20expects%20world%20semantics)[\[60\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,be%20recognized%20in%20this%20mode). If a user foolishly scripts something like echo ":host uname" | substrate, the :host will either be ignored or treated as an unknown command inside the world (most likely producing a “command not found: :host”). This prevents misuse in automation where everything should remain isolated. \- **Gating:** By default, :host is **disabled** even in interactive shells. We require the user to opt-in via a flag or config (for instance, substrate \--enable-host). If not enabled, when a user types a command starting with :host, we will not route to host. We can either output an error message (“Host escape is not enabled”) or treat it as a no-op. The decision (DR-04) was to gate it to prevent accidental discovery and use by automated wrappers[\[33\]](file://file_00000000a948722f938ff47e518d0244#:~:text=Option%20A%20,c%60%20or%20CI%2Fautomation%20modes)[\[61\]](file://file_00000000a948722f938ff47e518d0244#:~:text=,and%20uses%20the%20bypass%20in). In practice, an admin or advanced user can enable it when needed for troubleshooting. \- We will also ensure that any **sensitive operations via :host** are appropriately isolated. For example, if the user does :host vim somefile, that vim will run on host and could potentially write to host filesystem. That’s expected by the user (they explicitly used :host). We just need to not confuse it with world state. As such, host commands should probably run with the host’s current working directory equal to the project directory (or whatever it was before entering world). We will maintain a notion of host CWD separate from world CWD. Likely, at shell startup, host CWD \= project dir (since we call ensure\_session there without pivoting the host). If the user uses :host cd, we update host CWD accordingly. Host commands will execute with that as their working directory. This way :host pwd and :host ls etc. have a consistent context between host commands. This does mean if a user only does world cd (no :host cd), the host CWD remains at launch directory, which is fine because they’re not using it unless they do a :host command.

**Non-Interactive \-c/--command:** After this change, the semantics for a single-command execution with world enabled become consistent with “world-first” principles: \- If the world is enabled and available, the entire command given to \-c runs in the isolated world environment[\[12\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,be%20recognized%20in%20this%20mode). We will implement this by creating a world session (if not already) and using the world backend to execute the command (likely via backend.exec() with pty=false). This means even a simple built-in like substrate \-c "pwd" will actually be executed by /bin/sh inside the world and return the world’s working directory. This is important because if the project has an overlay, substrate \-c "pwd" should reflect that overlay’s perspective. \- We remove the previous “lightweight builtin” optimization for \-c. Formerly, the code might have directly handled \-c "cd X" or similar by doing it on host. That is no longer allowed when world is on[\[45\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,the%20operator%20expects%20world%20semantics). Instead, \-c "cd /dir && \<cmd\>" will be handled by the shell in the world. Notably, as per normal shell semantics, a cd in a one-shot command only applies to the following part of that command; once the shell exits, the effect is gone[\[62\]](file://file_00000000a948722f938ff47e518d0244#:~:text=,command%20effect%20%28standard%20shell%20semantics). That’s expected and acceptable. We just need to ensure we don’t have any leftover host side effect. \- The :host prefix is **invalid in \-c**. We won’t implement any handling for it; if someone tries substrate \-c ":host \<foo\>", it’s effectively an attempt to execute a program literally named “:host” in the world, which will result in an error message in stdout/stderr (command not found). This is fine; it’s unlikely a user would do that except by mistake. We will note in documentation that :host is only for interactive use and is not recognized in \-c mode[\[46\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,the%20operator%20expects%20world%20semantics)[\[63\]](file://file_00000000a948722f938ff47e518d0244#:~:text=Option%20B%20,command). \- If the world is disabled (via \--no-world or unsupported platform), then \-c behaves exactly as before (everything on host). In that case, we still might consider whether to keep builtin shortcuts. Given that our aim is to simplify semantics, we might drop builtin handling even in host mode \-c, but since it’s outside the scope of “world-first”, we can leave it as is for now. (The ADR explicitly limits changes to non-interactive mode to just removing host-builtins when world is enabled[\[64\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=Non,enforcement%20in)).

**Edge Cases:** \- **Empty Commands:** If user just hits enter, nothing should be sent to the world (just reprint prompt). \- **Commands starting with : that are not :host:** Substrate already has some commands like :demo-agent (for testing) or :pty (to drop into a subshell). We must make sure those still function or are adapted: \- :pty historically started a subshell *inside the world*. With the new model, the entire REPL is already effectively in a world subshell. We might deprecate :pty or make it simply open a nested shell if needed. But probably :pty becomes redundant. We should ensure :pty either is disabled or clearly documented to not be used. \- Other internal : commands (if any) should be considered. The decision\_register.md suggests :demo-agent was a thing for testing, presumably still fine. \- We will treat any unknown :xyz as before (some are for debug or not recognized). \- **Host and World Path Differences:** We should test scenarios like :host ls /home/substrate (where /home/substrate exists in world but not host) – obviously it will list host’s directory and likely not find that path. That’s expected. The key is users understand the context switch.

Overall, the rules are: **interactive mode uses world by default**, with :host as an escape for individual commands when enabled, and **non-interactive mode uses world by default as well**, with no escapes allowed in the command string. These rules will be clearly documented for users to avoid confusion. The consistency gained is that you won’t accidentally execute something on the host when you thought you were in an isolated environment – it will only happen if you explicitly ask for it.

## Compatibility and Rollout Strategy

* **Platform Focus – Linux First, macOS Next:** We will implement and validate this feature on Linux initially, where we have full control over namespaces and the world environment. Once stable, we will extend it to macOS (which uses Lima). On macOS, because the world is a VM, the same code should largely work – the substrate shell will connect to the Lima world-agent and spawn a shell in the VM. We’ll need to ensure the installer/bootstrapping sets up the Lima VM with a login shell available and possibly adjust any path differences (as noted). The ADR explicitly defers Windows support[\[56\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,specific%20design); the Windows rollout will come as a separate effort (Phase 5\) once we have ConPTY integration. So initially, on Windows, we may keep the current behavior (host fallback) or require WSL and treat it similarly to macOS’s VM. Given that Phase 5 plan (WSL integration) is completed[\[65\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/PHASE_5_ALWAYS_WORLD_WINDOWS_PLAN.md#L1-L9)[\[66\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/PHASE_5_ALWAYS_WORLD_WINDOWS_PLAN.md#L81-L89), by the time we release this, Windows likely uses WSL by default for any world features. But interactive PTY in Windows may still be experimental; we’ll clarify that Windows users might not get world-first REPL until a later version.

* **No Legacy Mode by Default:** We have chosen to **make this a hard cutover** to the new behavior without a long-term compatibility toggle[\[67\]](file://file_00000000a948722f938ff47e518d0244#:~:text=DR)[\[68\]](file://file_00000000a948722f938ff47e518d0244#:~:text=,for%20diagnosis%20during%20early%20adoption). That said, during rollout we will likely include a hidden escape hatch (an env var or \--legacy-repl flag) primarily for debugging or emergency fallback if a critical bug is discovered. This is similar to how we handled the async REPL rollout with a \--legacy-repl flag[\[69\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/tasks.json#L157-L165). The decision DR-03 opted for no official legacy mode[\[70\]](file://file_00000000a948722f938ff47e518d0244#:~:text=Option%20B%20,change%20as%20a%20hard%20cutover)[\[68\]](file://file_00000000a948722f938ff47e518d0244#:~:text=,for%20diagnosis%20during%20early%20adoption), so we will not advertise any, and plan to remove the fallback in a future release once confidence is high. For initial rollout, we might keep a behind-the-scenes switch (like SUBSTRATE\_LEGACY\_WORLD\_REPL=1) to disable persistent mode and use old behavior, solely for support purposes. This will not be documented to users, to discourage dependence on it.

* **Documentation and Comms:** We will update all relevant docs:

* User documentation to explain that as of this version, the shell is now “world-first”. We’ll highlight the differences: builtins now affect the world, the introduction of :host prefix, and removal of some previous quirks. The ADR’s Executive Summary provides a good operator-facing summary we can adapt for release notes[\[9\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,filesystem%20view%20the%20user%20sees)[\[71\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,Links). We’ll add examples of using :host and clarify that in scripts with \-c nothing special is needed (just that cd won’t persist beyond the command).

* Update docs/USAGE.md and docs/CONFIGURATION.md for any new flags (like \--enable-host).

* Add a section to troubleshooting: if something goes wrong with the world shell, how to recognize it (e.g., if user sees no output, maybe the marker parse failed – although that should be invisible – or if the world crashes, we should surface a warning).

* Ensure that our “security posture” docs emphasize that by default, everything is isolated unless :host is explicitly opted-in[\[61\]](file://file_00000000a948722f938ff47e518d0244#:~:text=,and%20uses%20the%20bypass%20in)[\[36\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=,NOT%20execute%20on%20the%20host).

* **Rollout Phases:** We might do a phased rollout:

* **Internal/Beta:** First release the feature in a canary or beta branch. Use it internally (dogfooding) to run common workflows. We expect to iron out any major issues (especially with persistent PTY interactions).

* Possibly release behind a feature flag (like an environment variable SUBSTRATE\_EXPERIMENTAL\_WORLD\_REPL) default-off for one version, so adventurous users can try it. However, given the decision to avoid long compat, we may skip this and go straight to making it default in a major/minor release.

* **General Release:** Once we’re confident, enable it by default in the next minor version release (and mention clearly in release notes that interactive shell behavior changed).

* Monitor user feedback, especially for any corner cases we didn’t anticipate (e.g., unusual shell usage patterns). If severe issues arise, instruct users to use the fallback env flag while we fix them.

* **Compatibility Considerations:**

* *Scripts and Tools:* Most automation uses substrate \-c or the non-interactive shim, which after this change will behave slightly differently only in obscure cases (like if someone relied on substrate \-c "cd X" changing something on host, which would have been odd). We don’t anticipate breakage in CI usage; in fact it fixes inconsistencies (e.g., previously substrate \-c "cd somedir && do\_something" might have failed if somedir was only present in overlay – now it will work properly in world).

* *Interactive Users:* Users might notice that their prompt or cwd indication changes (we might include the world id or a marker in the prompt to show isolation). We should make this user-friendly – perhaps prefix the prompt with something like \[world\] to remind them they’re in an isolated env. This is a minor UI tweak, but useful.

* *Old Workarounds:* Some advanced users used :pty bash as a workaround for the builtin issue[\[72\]](file://file_000000005ef471f5979d2b7bc6e54062#:~:text=%60cd%20,semantics%20once%20the%20PTY%20exits). With our changes, :pty bash is no longer needed – the default shell is already in world. We’ll document that the new default supersedes that usage. The :pty command might remain for launching an *additional* subshell (maybe inside the same world for debugging), but it’s lower priority.

* *Performance:* Starting a persistent shell in world has a one-time cost (forking a bash, etc.) but should be negligible (tens of milliseconds). It removes overhead on subsequent commands (no repeated agent handshake per command). If anything, users might find the REPL commands run faster now, especially on macOS (since going through vsock for each command had overhead, now a persistent connection streams data). We will validate performance but don’t foresee negative impact.

* **Windows Rollout:** Since Windows will require WSL and possibly the forwarder, we plan to release Linux/macOS first. For Windows, we might initially disable interactive world-first mode (or keep it experimental) until we finalize ConPTY support. The Phase 5 Windows plan suggests full parity (world-first on Windows via WSL) is achieved[\[65\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/PHASE_5_ALWAYS_WORLD_WINDOWS_PLAN.md#L1-L9)[\[73\]](https://github.com/atomize-hq/substrate/blob/655d3f431cad933fb6f087234429a1412f0fddd7/docs/project_management/_archived/phase_4-isolation/PHASE_5_ALWAYS_WORLD_WINDOWS_PLAN.md#L83-L91). If that’s ready, we will include Windows in the release, instructing Windows users to enable WSL2 and our substrate-forwarder. We will test on Windows thoroughly (ensuring that substrate.exe launching the WSL world yields identical trace outputs, that :host is properly handled – likely :host on Windows would just run the command in PowerShell or CMD on the Windows host, which is a bit different environment than bash; we need to clarify what host shell on Windows means. Possibly, we restrict :host on Windows or document it runs in a PowerShell context).

* **Regression Potential:** We will keep a close eye on any regressions in isolation (the worst would be a scenario where due to a bug, a command meant for world runs on host without user realizing). Our gating and require\_world checks are meant to prevent that. We’ll also ensure telemetry clearly indicates if something fell back to host (with warnings). After rollout, we will monitor telemetry from our own usage and maybe add temporary debug logs (e.g., log a warning if for any reason a world command is executed on host fallback).

In conclusion, the rollout strategy is to make world-first the standard behavior in a controlled manner: start with the robust Linux implementation, extend to macOS, provide clear opt-outs for emergencies but plan to remove them, and educate users on the new, improved semantics. This change will greatly improve shell usability and consistency with the isolation model, and with the above plan, we will deliver it with minimal disruption and strong validation.
